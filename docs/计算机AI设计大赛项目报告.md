# 学生睡眠管理系统

## 计算机 AI 设计大赛项目报告

项目名称：学生睡眠管理系统  
项目类型：AI + 教育健康管理应用  
技术方向：Web 应用、数据分析、智能建议、家校协同  
适用场景：班级睡眠健康管理、学生自我作息管理、老师预警监督、家长关注反馈

---

## 一、项目概述

学生睡眠管理系统是一款面向班级场景的 AI 辅助睡眠健康管理平台。系统围绕“学生自管、老师监督、家长关注”三方协同展开，通过每日睡眠打卡、睡眠质量评分、异常预警、可视化统计、AI 睡眠建议、成就激励和报表导出等功能，帮助学生建立良好作息习惯，帮助老师及时发现班级中的睡眠风险学生，帮助家长了解孩子近期睡眠状态。

与普通健康打卡系统不同，本项目不是简单记录数据，而是形成完整的闭环：

1. 学生起床后回顾填写睡眠数据。
2. 系统根据规则引擎自动计算质量分和风险状态。
3. 老师端实时查看班级概览、异常学生和趋势记录。
4. 家长端关注孩子睡眠摘要和严重预警。
5. AI 模块根据近期睡眠数据生成个性化改善建议。
6. 报表与可视化结果可用于班级管理和比赛展示。

系统当前采用 Vue 3 + Django 前后端分离架构，开发期使用 SQLite，支持通过环境变量切换 MySQL，并提供一键运行脚本和一键打包脚本，便于现场演示和作品提交。

## 二、选题背景与问题分析

### 2.1 背景

青少年睡眠问题已经成为学校和家庭共同关注的健康问题。学习压力、电子设备使用、作息不规律、周末补觉等因素都会影响学生睡眠质量。睡眠不足会进一步影响学习效率、情绪稳定、身体发育和课堂表现。

在实际班级管理中，老师通常只能通过课堂观察、家长反馈或个别谈话了解学生睡眠情况。这种方式存在信息滞后、数据不连续、责任边界不清晰等问题。家长虽然关心孩子睡眠，但也缺少一个既能保护学生隐私、又能及时了解异常情况的平台。

### 2.2 现有问题

当前学生睡眠管理主要存在以下痛点：

1. 数据采集不连续  
   学生是否早睡、睡眠是否稳定，往往缺少长期记录，老师难以判断趋势。

2. 异常发现滞后  
   严重熬夜、连续未打卡、周期性异常等问题通常在学生状态明显变差后才被发现。

3. 家校沟通分散  
   老师、家长和学生之间缺少统一协同入口，消息容易散落在聊天软件中。

4. 隐私边界模糊  
   睡眠日记可能包含学生心理状态和个人感受，如果直接开放给老师或家长，会造成隐私压力。

5. 缺少智能化建议  
   普通打卡系统只能记录数据，不能根据数据给出改善策略。

### 2.3 项目目标

本项目希望解决以上问题，建设一个可运行、可演示、可扩展的 AI 睡眠管理系统，实现以下目标：

1. 为学生提供简单、低负担的睡眠记录入口。
2. 为老师提供班级睡眠监管和异常预警能力。
3. 为家长提供孩子睡眠状态关注入口。
4. 用 AI 建议提升系统智能化程度。
5. 用可视化和报表提升作品展示效果。
6. 用清晰的隐私规则适配教育场景。

## 三、作品创新点

### 3.1 三角色协同闭环

系统不是单一学生工具，而是围绕学生、老师、家长三方协同设计。

学生负责记录和自我管理；老师负责班级监督和异常干预；家长负责关注孩子状态并接收预警。三方之间通过数据、通知和消息形成闭环。

### 3.2 AI + 规则引擎双模式

系统将睡眠评分逻辑独立为规则引擎，保证结果稳定、可解释、可测试。AI 模块在规则引擎的基础上，根据学生近期睡眠表现生成个性化建议。

当配置 DeepSeek API Key 时，系统可调用真实大模型；当无 Key 或网络异常时，系统自动使用 Mock 规则模板生成建议，保证比赛现场演示不会因外部接口失败而中断。

### 3.3 教育场景隐私保护

系统明确区分不同数据的隐私级别：

- 心情标签：老师和家长可见。
- 睡眠日记：默认仅学生可见。
- 日记分享：学生主动开启后，老师可见。
- AI 个人建议：仅学生本人可见。
- 班级排行：学生端匿名显示同学，老师端可查看真实姓名。

该设计兼顾监管需求和学生隐私，适合教育场景。

### 3.4 可解释的睡眠质量评分

系统评分不是黑盒算法，而是由睡眠时长、入睡时间和主观评分构成：

- 睡眠时长占 50 分。
- 入睡时间占 30 分。
- 主观评分占 20 分。

评分结果划分为健康、一般、异常、严重、未打卡五类状态。老师和学生都能理解分数来源，便于实际使用和比赛答辩说明。

### 3.5 演示友好的数据与报表能力

系统预置演示数据生成器，可生成 30 名学生、30 名家长、365 天睡眠记录、10 篇科普文章和 3 个典型案例：

- 严重熬夜学生。
- 完美作息学生。
- 周期性异常学生。

老师端可导出班级月报 Excel、单日概览 Excel 和个人学期 PDF，使作品更完整、更接近真实应用。

## 五、核心业务设计

### 5.1 起床后回顾打卡

系统采用“起床后回顾打卡”方式。学生早上填写昨晚入睡时间和今晨起床时间。相比睡前打卡，这种方式更容易得到完整睡眠时长，也更符合学生实际操作习惯。

### 5.2 跨天归属

睡眠记录的 `date` 字段表示“那晚所属日期”。例如学生在 2026-04-21 01:30 入睡，系统会将记录归属为 2026-04-20 晚。

规则如下：

- 00:00 至 04:59 入睡：归属前一天。
- 05:00 及以后入睡：归属当天。

### 5.3 补卡规则

学生允许补打最近 3 天内的记录。超过 3 天的记录视为 missed，不允许补打。这一规则兼顾灵活性和数据可信度。

### 5.4 修改规则

学生打卡后 2 小时内可修改记录。超过 2 小时后记录锁定。老师可代改记录，但必须填写修改原因，系统保存修改人和修改时间，形成审计追踪。

### 5.5 missed 自动标记

系统提供定时任务，每天 10:00 自动检查昨日未打卡学生，并插入 missed 记录。连续 2 天 missed 的学生自动进入重点关注状态。

### 5.6 周末阈值放宽

考虑学生周末作息相对宽松，系统对周五和周六晚入睡阈值放宽 30 分钟。工作日晚健康线为 23:30，周末晚健康线为 00:00。

## 六、AI 与算法设计

### 6.1 睡眠评分规则

系统使用 100 分制计算睡眠质量分。

| 维度 | 权重 | 说明 |
|---|---:|---|
| 睡眠时长 | 50% | 以 8 小时为目标，7-9 小时为较优 |
| 入睡时间 | 30% | 工作日 23:30 前较优，周末放宽至 00:00 |
| 主观评分 | 20% | 学生填写 1-5 分 |

质量分对应状态：

| 分数 | 状态 |
|---:|---|
| 85-100 | 健康 normal |
| 70-84 | 一般 warning |
| 50-69 | 异常 abnormal |
| 0-49 | 严重 severe |
| 未打卡 | missed |

### 6.2 AI 建议生成

学生触发 AI 建议时，系统收集近 7 天数据：

- 平均质量分。
- 平均睡眠时长。
- 连续打卡天数。
- 近期最差状态。
- 最近入睡时间。

系统将这些数据整理成 prompt，并调用 DeepSeek 生成建议。建议内容控制在短文本范围内，强调可执行性，例如提前上床、减少睡前手机、固定作息、睡前放松等。

### 6.3 Mock 兜底机制

比赛现场经常存在网络不稳定、API Key 不可用或额度受限等风险。为保证系统可稳定演示，本项目设计了 Mock 兜底：

1. 如果未配置 `DEEPSEEK_API_KEY`，直接返回规则模板建议。
2. 如果 API 调用超时或失败，也返回规则模板建议。
3. 前端展示结果时标记是否为 Mock。

这种设计体现了 AI 应用工程化落地思维：既利用大模型能力，又保证系统可用性。

## 七、数据库设计

系统核心数据表如下：

| 表名 | 作用 |
|---|---|
| User | 用户账号、角色、邮箱预警开关 |
| ClassRoom | 班级、老师、班级邀请码 |
| StudentProfile | 学生档案、目标睡眠、风险等级、家长邀请码 |
| TeacherProfile | 老师档案 |
| ParentProfile | 家长档案和孩子绑定 |
| SleepRecord | 睡眠记录、评分、状态、审计字段 |
| Notification | 站内通知、消息、预警 |
| Article | 科普文章 |
| Achievement | 成就定义 |
| StudentAchievement | 学生成就解锁记录 |
| AIAdvice | AI 建议历史 |

其中 `SleepRecord` 是核心表，关键字段包括：

- `student_id`：所属学生。
- `date`：那晚所属日期。
- `bedtime`：入睡时间。
- `wake_time`：起床时间。
- `duration_minutes`：睡眠时长。
- `quality_score`：质量分。
- `status`：状态。
- `mood_tag`：心情标签。
- `diary`：睡眠日记。
- `modified_by`：修改人。
- `modified_reason`：修改原因。
- `modified_at`：修改时间。

系统对 `student + date` 建立唯一约束，避免同一学生同一天重复打卡。

## 八、系统实现

### 8.1 后端实现

后端使用 Django 按业务模块拆分：

```text
apps/users           用户、班级、邀请码、角色档案
apps/sleep           睡眠记录、统计、排行、导出入口
apps/notifications   通知、消息、预警、定时任务
apps/articles        科普文章
apps/achievements    成就徽章
apps/ai              AI 建议与 DeepSeek 客户端
utils/score_engine   睡眠评分引擎
utils/exporters      Excel/PDF 导出
utils/seed_data      演示数据生成
utils/scheduler      定时任务调度
```

后端接口采用 DRF 编写，认证使用 JWT。权限控制通过角色守卫实现，确保学生、老师、家长和管理员只能访问对应数据。

### 8.2 前端实现

前端使用 Vue 3 和 Element Plus 构建界面，使用 Vue Router 实现角色路由。

学生、老师、家长和管理员分别有独立布局：

```text
layouts/StudentLayout.vue
layouts/TeacherLayout.vue
layouts/ParentLayout.vue
layouts/AdminLayout.vue
```

API 请求统一由 axios 封装，自动携带 JWT，并在 token 过期时尝试刷新。页面按角色放在 `views/student`、`views/teacher`、`views/parent` 和 `views/admin` 目录下。

### 8.3 可视化实现

系统使用 ECharts 展示统计图：

- 学生近 7 天质量分柱状图。
- 老师端今日打卡饼图。
- 学生年度睡眠热力图。
- 学生趋势表格与状态颜色。

通过可视化，比赛评委可以快速理解系统价值和数据变化。

### 8.4 报表实现

系统提供三类报表：

1. 班级月报 Excel：展示全班每日质量分和状态颜色。
2. 单日班级概览 Excel：展示某天全班睡眠明细。
3. 个人学期 PDF：展示学生学期睡眠摘要和近期明细。

报表能力让系统更接近真实校园管理应用。

## 十一、应用价值

### 11.1 教育价值

系统可以帮助学校班级建立科学、持续的睡眠管理机制，让老师从被动观察转向主动预警，让学生从被动被管转向自我管理。

### 11.2 健康价值

通过持续记录、评分反馈和 AI 建议，学生可以更直观地理解睡眠习惯与日常状态之间的关系，从而逐步改善作息。

### 11.3 管理价值

老师可以通过班级概览、异常筛选和报表导出快速掌握全班情况，减少人工统计成本。

### 11.4 推广价值

当前系统面向单班级，但架构上可扩展到年级、学校或区域级健康管理平台。后续也可以接入智能手环、移动端和更多健康指标。

## 十二、风险与改进方向

### 12.1 当前风险

1. 数据依赖学生手动填写，存在主观误差。
2. AI 建议质量依赖输入数据质量和模型能力。
3. 邮件推送需要 SMTP 配置，比赛现场默认以站内通知为主。
4. 当前主要面向单班级，复杂组织结构仍需扩展。

### 12.2 后续改进

1. 接入智能手环或手机健康数据，提高数据准确性。
2. 增加移动端或小程序，提高学生打卡便利性。
3. 引入更长周期趋势分析，提升 AI 建议个性化程度。
4. 增加年级主任、学校管理员等角色。
5. 增加更完整的操作日志和隐私授权记录。

## 十三、项目总结

学生睡眠管理系统以 AI 辅助教育健康管理为主题，将睡眠打卡、数据评分、异常预警、家校协同、隐私保护、智能建议和报表导出整合到一个完整 Web 平台中。系统既具备明确的现实应用场景，也具备适合计算机 AI 设计大赛展示的技术亮点。

本项目的核心价值在于：用可解释规则保证基础判断可靠，用 AI 建议提升服务智能化，用三角色协同完成真实业务闭环，用一键运行和演示数据保证作品可展示、可评审、可复现。

从当前实现情况看，系统已经完成主要业务功能和工程化交付能力，能够作为计算机 AI 设计大赛的完整参赛作品进行展示。
